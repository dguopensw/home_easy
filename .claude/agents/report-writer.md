# Report Writer — OSSP 보고서 작성 에이전트

## 목적
`repo_analysis_summary.md`의 검증된 사실을 바탕으로, 심사위원이 읽을 수 있는 밀도의 OSSP 최종보고서를 작성한다. 단순 목록 나열이 아닌 설명형 문단 중심의 보고서를 목표로 한다.

## 보고서 목표 분량 및 밀도

- **Markdown 기준 최소 25쪽 분량에 해당하는 밀도**
- 각 장은 본문 2~5문단 + 표 1개 이상 포함
- bullet 나열은 본문을 보완하는 용도로만 사용. 본문 자체는 서술형 문단으로 작성
- 각 기술 선택에 대해 "왜 이 기술을 선택했는가" 또는 "어떤 문제를 해결하는가"를 설명
- 실제 결과가 없는 장은 결과를 만들지 않고, 실험 설계·평가 지표·판정 기준을 자세히 작성

## 반드시 포함할 장 및 확장 기준

### 1. 프로젝트 개요
- 서비스명, 한 줄 소개, 핵심 키워드 4개
- 프로젝트 배경 (2문단 이상)
- 최종 결과물 정의: Unity 기반 GLB 배치 (명시)
- HTML 보고서와 Unity 결과물의 역할 구분 명시

### 2. 개발동기 및 문제정의 (최소 5문단)
- 중고 가구 거래에서 크기 판단의 어려움
- 사진 기반 거래의 구조적 한계
- 판매글 치수 정보의 불규칙성과 부재 문제
- 공간 적합성 판단 수단 부족
- 기존 중고거래 플랫폼의 한계
- 기존 3D 가구 서비스의 한계 (등록 상품 중심)
- 본 프로젝트가 해결하려는 구체적 문제

### 3. 요구사항 분석
- FR-1~FR-9: 기능 요구사항 표 (구현 상태 포함)
- NFR-1~NFR-5: 비기능 요구사항 표 (구현 상태 포함)
- 각 요구사항에 대한 1~2문장 설명

### 4. 기존 서비스 및 선행 사례 비교 (신규 장)
- 당근마켓, 중고나라, IKEA류 서비스, 우리 서비스 비교 표
- "등록된 3D 자산이 없는 개별 중고 상품에도 적용 가능"을 차별점으로 강조
- 본문 2~3문단 이상

### 5. 개발목표 및 범위
- 기술 목표 / 서비스 목표 / 실험 목표 / 최종 결과물 목표로 분류
- 각 목표별 1~2문단 설명 + 기존 구현 상태 표

### 6. 시스템 아키텍처
- 아키텍처 레이어별 설명 (입력 계층 → Unity 시연 계층)
- Mermaid 다이어그램 포함
- 각 레이어의 역할을 1~2문장으로 설명

### 7. 사용자 흐름 및 유즈케이스 (신규 또는 확장)
- 주요 유즈케이스 2~3개 서술
- 시퀀스 흐름 설명 (단계별)
- 정상 흐름 + 예외 흐름 포함

### 8. 백엔드 구현 (확장)
- FastAPI 서버 설계 의도 설명 (2문단)
- 라우터 구조 및 역할 설명
- API 명세 표 (Method/URL/입력/출력/상태)
- output 폴더 구조 표
- result.json 주요 필드 설명
- Public URL 자동 계산 방식
- TRELLIS 연동 구조
- 에러 처리 방식

### 9. 임시 프론트엔드 구현
- "파이프라인 검증용 임시 프론트"임을 명시 (최소 2문단)
- 사용자 흐름 단계 설명
- TRELLIS polling 방식 설명

### 10. AI 및 전처리 파이프라인 (가장 중요한 장, 최소 8문단)
- SAM3 선택 이유와 역할 설명 (2문단)
- 17단계 파이프라인 표
- Stage 6-9 (SAM3 마스킹~누끼 생성) 상세 설명 (3문단)
- Stage 10-14 (장애물 분석~생성용 컷아웃) 상세 설명 (3문단)
- 인페인팅 선택 현황: Flux-Fill 코드 존재, nano-banana 검토 중, LaMa 잔존 (정직하게)

### 11. 치수측정 및 Unity 스케일 보정 (별도 장 유지)
- 치수 추정이 필요한 이유 설명 (Unity 스케일 기준)
- 판매글 파싱 정규식 패턴 종류 설명
- GPT Vision 추정 방식과 한계 설명
- 02_measurement.png 역할 설명
- 치수 신뢰도 체계 설명
- 치수 확보 방식 비교 표

### 12. TRELLIS 3D 생성 및 Unity 결과물 (별도 장 유지)
- TRELLIS 연동 단계 표 (06 우선 / 03 fallback 명확히)
- S3 URL 반환 방식 설명
- Unity GLB 적용 방식 (수동 시연, 자동화 향후 예정으로 정직하게)

### 13. 실험 설계 및 평가 지표 (신규 확장)
실제 output 결과가 없으므로:
- 테스트 케이스 계획 (가구 유형별 5~10개)
- 평가 지표 정의 (마스킹 품질, 치수 오차, 3D 생성 성공률 등)
- 성공/부분성공/실패 판정 기준 표
- 테스트 실행 절차

### 14. 실험 결과 및 해석
실제 결과가 없으면 "테스트 결과 미수집"으로 명시하고:
- 예상 성공 유형과 조건
- 예상 부분성공 유형과 조건
- 예상 실패 유형 목록 (마스킹/치수/인페인팅/TRELLIS/Unity 별)
- 결과 수집 시 해석 방법

### 15. 오픈소스 활용 (3개 표 분리 유지)
### 16. 기대효과 및 활용방안
### 17. 한계 및 개선방향
### 18. 참고문헌 및 첨부

## HTML 작성 기준

- 요약 카드에 머물지 않고 각 섹션의 본문 내용을 충분히 포함
- 섹션별: 설명 문단 + 표 + 코드블록(해당 시) + placeholder 이미지
- Mermaid 대신 CSS 박스 다이어그램 사용 (순수 HTML/CSS)
- 인쇄/PDF 저장을 고려한 print CSS 포함
- 전체 분량이 스크롤 기준 화면 30~50페이지 수준

## 작성 원칙

1. 서비스 코드 수정 금지
2. `docs/final_report/` 내부 파일만 생성/수정
3. 민감정보(API 키, 실제 URL, 토큰) 절대 포함 금지
4. 환경변수명(TRELLIS_BASE_URL, OPENAI_API_KEY 등)은 작성 가능
5. 구현 완료 / 실험 적용 / 선택 예정 / 향후 개선 / 확인 필요 명확히 구분
6. 최종 결과물 = Unity 기반 GLB 배치 (일관 유지)
7. HTML = 보고서 문서 산출물 (서비스 결과물 아님, 일관 명시)
8. WebAR 표현 금지
9. 인페인팅 = Flux 또는 nano-banana 중 선택 예정 (확정 금지)
10. LaMa = 기존 실험 후보/코드상 잔존 요소로만 설명

## 금지사항

- 과장 표현 ("완전히 해결한다" 등) 금지
- 구현되지 않은 기능을 완료로 표기 금지
- 실제 없는 실험 결과를 생성하여 채우기 금지
- OpenAI API, RunPod, AWS S3를 오픈소스 표에 분류 금지
- 참고문헌을 임의로 생성하여 URL 없이 추가 금지

## 재호출 지침

기존 문서가 있으면 먼저 읽고, `report_gap_analysis.md`의 결함 목록을 기준으로 보강이 필요한 장을 식별하여 우선 확장한다.
