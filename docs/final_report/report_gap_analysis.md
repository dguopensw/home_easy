## 1. 현재 보고서의 문제점

### 1-1. 본문 밀도 부족

| 장 번호 | 장 제목 | 현황 | 문제점 |
|---------|---------|------|--------|
| 1장 | 프로젝트 개요 | 짧은 bullet 목록 위주 | 개발동기가 1문단이며 설득력 부족 |
| 2장 | 개발목표 및 범위 | 표만 있음 | 각 목표별 설명 문단 없음 |
| 3장 | 최종 결과물 소개 | 흐름 설명 수준 | 아키텍처 레이어 설명 없음, Mermaid 다이어그램 없음 |
| 4장 | 백엔드 구현 | API 명세 표 포함 | output 폴더 구조, result.json 설명 없음 |
| 5장 | 임시 프론트엔드 | 2문단 있음 | TRELLIS polling 방식 미설명 |
| 6장 | AI 파이프라인 | 17단계 표 포함 | Stage별 "왜 필요한지" 설명 3문단 이상 부족 |
| 7장 | 치수측정 | 독립 장으로 존재 | 신뢰도 체계 및 확보 방식 비교 표 없음 |
| 8장 | TRELLIS 3D 생성 | 연동 단계 표 포함 | S3 URL 반환 방식, Unity 수동 적용 이유 설명 없음 |
| 9장 | 실험 결과 | "확인 필요"로만 채워짐 | 실험 설계, 평가 지표, 판정 기준 없음 |
| 10장 | 오픈소스 | 3개 표 있음 | 라이선스 전부 "확인 필요", 일부 항목 분류 오류 가능성 |
| 11장 | 기대효과 | 목록 나열 수준 | 설명 문단 없음 |
| 12장 | 한계 | 목록 있음 | 각 한계별 개선 방향 연결 없음 |
| 13장 | 참고문헌 | 임시 항목 | URL 없는 항목 다수 |

### 1-3. 기술 설명 품질 문제

| 항목 | 현황 | 필요한 내용 |
|------|------|------------|
| SAM3 선택 이유 | "SAM3 기반 마스킹"으로만 서술 | 왜 SAM3인가, 기존 방식 대비 장점 |
| 인페인팅 파이프라인 내 역할 | "Flux 또는 nano-banana 후보"만 명시 | 어떤 조건에서 필요한지, 입출력 이미지 설명 |
| 치수 신뢰도 체계 | 언급 없음 | listing_text 우선 / vision_estimate 폴백 이유 |
| TRELLIS 입력 우선순위 | 언급 없음 | 06 우선, 03 fallback 이유 설명 |
| Unity 수동 적용 이유 | "자동화 향후 검토"만 언급 | 현재 수동인 이유, 자동화 시 필요한 것 |

### 1-4. HTML 보고서 문제

| 항목 | 현황 | 필요한 내용 |
|------|------|------------|
| 분량 | 스크롤 기준 약 10~15 화면 | 30~50 화면 목표 |
| 구조 | 카드형 요약 위주 | 각 섹션에 설명 문단 + 표 + 시각화 |

| 실험 설계 섹션 | 없음 | 테스트 케이스, 평가 지표, 판정 기준 |
| 개발동기 | 없음 또는 짧음 | 5문단 이상 별도 섹션 |
| 요구사항 분석 | 없음 | FR/NFR 표 2개 |
| 기존 서비스 비교 | 없음 | 비교 표 + 차별점 설명 |

## 3. 우리 보고서에 맞춘 보강 방향

### 3-1. 개발동기 보강 (최소 5문단)

- 중고 가구 거래의 규모와 사진 기반 거래의 구조적 한계
- 판매글 치수 정보의 불규칙성과 공백 문제
- 공간 적합성 판단 수단의 부족 (구매자 시점)
- 기존 중고거래 플랫폼의 한계 (당근마켓, 중고나라)
- 기존 3D 가구 서비스의 한계 (등록 상품 중심, 임의 중고 상품 불가)
- 본 프로젝트가 해결하려는 구체적 문제

### 3-2. 기술 설명 보강 방향

**SAM3 선택 이유:**
- 단순 배경 제거(rembg) 대비 마스킹 정밀도
- 프롬프트 기반 영역 지정 가능
- 점/박스 프롬프트로 가구 영역을 명확하게 타겟팅 가능

**인페인팅 필요성:**
- 가구 주변에 다른 물체(장애물/오염물)가 있을 때
- 누끼 이미지에 불필요한 요소가 남아 3D 품질에 영향을 미칠 때
- Flux-Fill이 코드에서 실제 호출됨, nano-banana는 검토 중 (정직하게 유지)

**치수 신뢰도 체계:**
- listing_text 파싱: 가장 신뢰도 높음
- GPT Vision 추정: 폴백. 단일 이미지에서 절대 치수 추정의 한계 존재
- 치수 없이 진행: 스케일 보정 제한적

**TRELLIS 06/03 우선순위:**
- 06_generation_cutout.png: 인페인팅 후 최종 컷아웃, 가장 깨끗한 배경 제거 상태
- 03_final_cutout.png: 인페인팅 미적용 fallback, 06이 없거나 인페인팅 실패 시 사용

### 3-3. 실험 설계 보강 방향 (실제 결과 없을 때)

- 테스트 케이스: 소파, 의자, 책상, 침대, 옷장 각 2개씩 (총 10개)
- 평가 지표:
  - 마스킹 품질: IoU 또는 주관적 5점 척도
  - 치수 오차: (추정 치수 - 실제 치수) / 실제 치수 × 100%
  - 3D 생성 성공률: TRELLIS 서버 응답 기준
  - Unity 배치 품질: 스케일 시각 검증
- 성공/부분성공/실패 판정 기준
- 테스트 실행 절차

### 3-4. 자료 체크리스트 보강

| 항목 | 취득 방법 | 우선순위 |
|------|----------|---------|
| 파이프라인 실행 결과 이미지 (01~06) | 백엔드 실행 후 output/ 캡처 | 높음 |
| Unity 배치 결과 캡처 | GLB import 후 스케일 보정 캡처 | 높음 |
| 성공/실패 비교 이미지 | 여러 가구 유형 실행 | 높음 |
| 임시 프론트 화면 캡처 | 백엔드 실행 후 브라우저 캡처 | 중간 |
| 시스템 아키텍처 다이어그램 | draw.io 또는 Figma | 중간 |
| TRELLIS 서버 상태 확인 | RunPod 대시보드 | 중간 |

## 4. 확장 후 목표 목차 (18장)

```
# 오픈소스 프로젝트 최종보고서

1. 프로젝트 개요
2. 개발동기 및 문제정의           ← 신규 독립 장 (5문단 이상)
3. 요구사항 분석                  ← 신규 (FR/NFR 표 2개)
4. 기존 서비스 및 선행 사례 비교   ← 신규 (비교 표 + 차별점)
5. 개발목표 및 범위               ← 확장 (목표별 설명 문단 추가)
6. 시스템 아키텍처                ← 확장 (레이어 설명 + Mermaid)
7. 사용자 흐름 및 유즈케이스       ← 독립 장 분리
8. 백엔드 구현                   ← 확장 (output 구조, result.json)
9. 임시 프론트엔드 구현            ← 유지
10. AI 및 전처리 파이프라인        ← 확장 (SAM3 이유, Stage별 설명)
11. 치수측정 및 Unity 스케일 보정  ← 확장 (신뢰도 체계, 비교 표)
12. TRELLIS 3D 생성 및 Unity 결과물 ← 확장 (06/03 이유, 수동 이유)
13. 실험 설계 및 평가 지표         ← 신규 독립 장
14. 실험 결과 및 해석              ← 분리 (결과 미수집 명시 + 예상 유형)
15. 오픈소스 활용                  ← 유지 (라이선스 보강)
16. 기대효과 및 활용방안            ← 확장 (설명 문단)
17. 한계 및 개선방향               ← 확장 (개선방향 연결)
18. 참고문헌 및 첨부               ← 유지 (URL 확인 필요 명시)
```
